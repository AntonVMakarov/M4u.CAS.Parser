# Математические, логические и сравнительные операторы для парсера

Версия: 2026-05-22. Это суженная таблица для лексера/парсера математических выражений: арифметика, анализ, множества, сравнения и логика. Из предыдущей широкой таблицы убраны присваивания, правила замены, строковые/служебные/программные операторы, кроме случаев, где они реально конфликтуют с математической записью.

## Как читать таблицу

- `Приоритет` — практическая шкала для Pratt parser / precedence climbing: большее число связывает сильнее. Это не единственный стандарт; для строгой совместимости с конкретной CAS надо брать её таблицу приоритетов.

- `binder` — оператор, который связывает переменную и тело выражения: `∫`, `∑`, `lim`, `∀`, `∃`. Такие формы обычно лучше парсить отдельным правилом области захвата, а не только числом приоритета.

- **✅ уникален** — токен сам по себе обычно однозначен. Семантика всё равно может зависеть от типов операндов.

- **⚠️ похож/конфликтен** — токен совпадает с другим оператором, является префиксом более длинного токена, визуально похож на другой символ или требует контекста для арности.


## Рекомендуемые уровни приоритета

| Уровень | Смысл |
|---:|---|
| 1000 | Явные скобки, matchfix-операторы, вызов/индексация |
| 990–965 | Индексы, postfix-операторы, штрихи, factorial |
| 950 | Степени, корни, superscript |
| 930 | Унарные prefix-операторы, производные, ∇ |
| 910–900 | Интегралы, суммы, произведения, пределы как binder |
| 880–870 | Умножение, деление, dot/cross/tensor |
| 840–820 | Пересечение/объединение множеств, симметрическая разность |
| 830 | Сложение, вычитание, разность множеств |
| 700 | Сравнения и отношения; допускайте цепочки `a < b <= c` |
| 650–560 | Логика: NOT, AND, XOR, OR, IMPLIES, IFF |
| 540–520 | Кванторы и отношения выводимости |
| 120 | Низкоприоритетные разделители внутри set-builder/proof notation |

## Самые опасные места для токенизации

1. **Maximal munch обязателен**: сначала распознавайте самые длинные варианты `!==`, `===`, `<=`, `>=`, `!=`, `==`, `&&`, `||`, `!!`, `**`, `<<`, `>>`, `<=>`, а уже потом их префиксы.

2. **Один символ — несколько арностей**: `+`, `-`, `±`, `∓`, `!`, `|`, `∧`, `^` требуют контекста: начало выражения/после оператора → prefix; после операнда → postfix или infix.

3. **ASCII и Unicode лучше не смешивать без нормализации**: `-` и `−`, `*` и `×`, `.` и `·`, `<=` и `≤`, `|` и `∣` удобно хранить как разные raw-токены, но маппить на один AST-оператор только после лексинга.

4. **Вертикальная черта `|` — самый конфликтный символ**: модуль, норма, OR, делимость, such-that, условная вероятность. Для серьёзного парсера лучше различать режимы: expression, set-builder, probability, logic.

5. **Сравнения лучше парсить цепочкой**, а не обычным бинарным AST: `a < b <= c != d` должно превращаться в relation-chain или conjunction, а не в `(((a<b)<=c)!=d)`.

6. **Binder-операторы требуют области захвата**: `∫`, `∑`, `lim`, `∀`, `∃` почти всегда нуждаются в отдельном синтаксическом правиле, особенно в 2D/LaTeX/Wolfram-подобной записи.


## Минимальные правила лексера

- Разделите лексинг на `raw token` и последующее `semantic token`: например, `<=` и `≤` могут давать общий оператор `LessEqual`, но raw-форма должна сохраняться для диагностики.

- Держите отдельный словарь Unicode-операторов и отдельный словарь ASCII-мультисимвольных операторов. ASCII-словарь сортируйте по длине по убыванию.

- Для `-`, `+`, `!`, `|`, `^`, `_`, `.` применяйте не только лексические, но и синтаксические правила. Их нельзя полностью разобрать регуляркой.

- Для неявного умножения включите отдельную фазу после токенизации: вставляйте виртуальный `Times` между совместимыми токенами, например `2 x`, `2(x+1)`, `(x+1)(y+1)`, но не внутри идентификатора `xy`.
