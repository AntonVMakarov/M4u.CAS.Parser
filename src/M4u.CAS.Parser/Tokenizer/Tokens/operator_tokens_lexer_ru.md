# Операторные лексемы для первой стадии парсинга: строка → массив токенов

Версия: 2026-05-22

Этот файл — практическая таблица именно для **лексического анализа** математических выражений. На этой стадии лучше не решать, является ли `-` унарным или бинарным, `!` — факториалом или отрицанием, а `|` — модулем, OR или делимостью. Лексер должен сохранить исходную лексему и позицию, а синтаксический парсер уже разберёт роль токена по контексту.

## Что сознательно убрано

Из таблицы убраны **binder-операторы**, потому что они связывают переменную/индекс и область действия, а не просто дают один токен-оператор:

- суммы/произведения/копроизведения с областью: `∑`, `∏`, `∐`;
- интегралы и контурные интегралы: `∫`, `∮`, `∯`, `∰`, `∱`, `∲`, `∳`;
- пределы и похожие формы: `lim`, `limsup`, `liminf`, `sup`, `inf` в binder-нотации;
- кванторы: `∀`, `∃`, `∄`;
- lambda/mapsto-формы, если они связывают параметры: `λ`, `↦`.

Если такие конструкции понадобятся, лучше добавлять их отдельным синтаксическим правилом: «binder + список связанных переменных + тело/область», а не пытаться обработать как обычный инфиксный/префиксный оператор.

## Легенда статуса

- **✅ уникален** — лексему обычно можно однозначно выделить как отдельный токен. Это не значит, что её математический смысл всегда один и тот же.
- **⚠️ похож/конфликтен** — лексема является префиксом более длинной лексемы, визуально похожа на другой символ, конфликтует с идентификаторами/числами/строками или имеет несколько ролей уже на уровне токенизации.

Рекомендуемая форма токена для первой стадии:

```ts
type Token =
  | { kind: "number"; raw: string; start: number; end: number }
  | { kind: "identifier"; raw: string; start: number; end: number }
  | { kind: "operator"; raw: string; canonical?: string; start: number; end: number }
  | { kind: "delimiter"; raw: string; start: number; end: number }
  | { kind: "keyword"; raw: string; canonical: string; start: number; end: number };
```

Главный принцип: **одна найденная лексема → один токен**, а `canonical` можно использовать для сведения, например, `≤` и `<=` к одному внутреннему значению без потери `raw`.

| `∨`; `\|\|`; `\|`; `or` | логическое ИЛИ / join | ⚠️ похож/конфликтен | `\|\|` конфликтует с нормой `\|\|x\|\|`; `\|` конфликтует почти со всем; `or` конфликтует с идентификаторами. | Сначала `\|\|`, потом `\|`; Unicode `∨` безопаснее; `or` — keyword-only. |
| `⊻`; `xor`; `^` | исключающее ИЛИ | ⚠️ похож/конфликтен | `^` почти всегда конфликтует со степенью; `xor` — слово-идентификатор; `⊻` безопаснее. | Не канонизировать `^` в XOR на лексической стадии; выдавать `OP("^")` и решать смысл грамматикой. |
| `⊼`; `nand` | NAND | ✅ уникален | `nand` как слово конфликтует с идентификаторами, но Unicode `⊼` уникален. | Unicode напрямую; слово — только keyword с границами. |
| `⊽`; `nor` | NOR | ✅ уникален | `nor` как слово конфликтует с идентификаторами, но Unicode `⊽` уникален. | Unicode напрямую; слово — keyword с границами. |
| `→`; `⇒`; `⟹`; `->`; `=>` | импликация / стрелка | ⚠️ похож/конфликтен | `->` начинается с `-`; `=>` начинается с `=`; в CAS `->` часто правило замены, а не импликация. | Сначала длинные стрелки ASCII; Unicode-стрелки безопаснее. Смысл “правило/импликация” оставить парсеру. |
| `←`; `⇐`; `⟸`; `<-` | обратная импликация / стрелка влево | ⚠️ похож/конфликтен | `<-` начинается с `<`; может конфликтовать со сравнением и присваиванием в некоторых языках. | Сначала `<-`, потом `<`; Unicode-стрелки безопаснее. |
| `↔`; `⇔`; `⟺`; `<->`; `<=>` | эквивалентность / iff | ⚠️ похож/конфликтен | `<->` и `<=>` имеют префикс `<`; `<=>` также включает `<=` и `=>`. | Longest match: `<=>`, `<->`, затем `<=`, `<`, `=>`, `=`. Unicode безопаснее. |
| `⊢`; `⊬`; `⊨`; `⊭`; `⊩`; `⊮`; `⊧`; `⊭` | выводимость / следование / модели | ✅ уникален | Лексически однозначны; относятся скорее к метаязыку логики. | Токенизировать напрямую; не включать кванторы `∀`, `∃` как обычные операторы, если binders исключены. |
| `⊤`; `⊥`; `true`; `false`; `True`; `False`; `T`; `F` | логические константы | ⚠️ похож/конфликтен | `T`, `F`, `true`, `false` могут быть идентификаторами; `⊥` также перпендикулярность. | Слова распознавать как ключевые только по конфигурации языка; Unicode-константы безопаснее. |
| `∴`; `∵` | следовательно / потому что | ✅ уникален | Лексически однозначны; чаще разделители рассуждения, чем операторы выражений. | Токенизировать напрямую или разрешать только в proof-mode. |
| `mod`; `%`; `modulo` | остаток / сравнение по модулю | ⚠️ похож/конфликтен | Слова конфликтуют с идентификаторами; `%` конфликтует с процентом и CAS-синтаксисом. | Если поддерживается `mod`, делать keyword-only. `%` лучше не включать без явной политики. |
| `div`; `gcd`; `lcm` | текстовые математические операторы/функции | ⚠️ похож/конфликтен | Это может быть обычный идентификатор функции, а не оператор. | На первой стадии лучше выдавать `IDENT`, а не `OP`, если вы не хотите жёстко фиксировать словарь функций. |


## Минимальный алгоритм токенизации

### 1. Не делайте `-1` одним числом

Для математического парсера обычно лучше получать:

```text
-1        → OP("-"), NUMBER("1")
a-b       → IDENT("a"), OP("-"), IDENT("b")
a--b      → IDENT("a"), OP("-"), OP("-"), IDENT("b") или OP("--") только если вы явно поддерживаете `--`
```

Исключение — показатель экспоненты в числе:

```text
1e-3      → NUMBER("1e-3")
1.2E+10   → NUMBER("1.2E+10")
```

Минус/плюс после `e` или `E` можно считать частью числа, потому что это не оператор выражения, а часть числового литерала.

### 2. Используйте longest match / maximal munch

Многие опасные операторы отличаются только длиной. Поэтому операторный словарь надо искать не регулярками в случайном порядке, а через trie или список, отсортированный по длине убывающе.

Пример обязательного порядка:

```text
<=>, <->, <=, <>, <<, <-, <
!==, !=, !!, !
===, ==, =>, =
:=, ::, :,
**, *,
||, |,
&&, &,
... , .. , .
```

Псевдокод:

```ts
while (pos < input.length) {
  if (isWhitespace(input[pos])) { skip(); continue; }

  const num = readNumber(input, pos);       // включая 1e-3, но не обычный -1
  if (num) { emit(num); pos = num.end; continue; }

  const ident = readIdentifier(input, pos); // Unicode XID_Start/XID_Continue желательно
  if (ident) {
    emitKeywordOrIdentifier(ident);         // and/or/not/in только на границе слова
    pos = ident.end;
    continue;
  }

  const op = operatorTrie.longestMatch(input, pos);
  if (op) { emitOperatorOrDelimiter(op); pos += op.length; continue; }

  error(`Unknown character at ${pos}`);
}
```

### 3. Не решайте роль `|` в лексере

`|` может быть:

```text
|x|              абсолютная величина
||x||            норма
a | b            делимость или OR
{x | P(x)}       “такие, что”
P(A | B)         условная вероятность
```

На первой стадии разумнее выдать `BAR`/`OP("|")` и оставить решение парсеру. Если хочется облегчить жизнь парсеру, можно иметь отдельный Unicode-токен `‖` для нормы и `∣` для делимости, но ASCII `|` всё равно должен оставаться контекстным.

### 4. Ключевые слова распознавайте только по границам идентификатора

`and`, `or`, `not`, `in`, `xor`, `mod` не должны срабатывать внутри имён:

```text
origin      → IDENT("origin"), а не OP("or") + IDENT("igin")
notion      → IDENT("notion"), а не OP("not") + IDENT("ion")
xorValue    → IDENT("xorValue"), а не OP("xor") + IDENT("Value")
```

Если пользователь может называть переменные `and` или `or`, сделайте ключевые слова отключаемыми или требуйте экранирование идентификаторов.

### 5. Не используйте NFKC вслепую

Unicode-нормализация полезна, но опасна. Например, superscript/subscript-символы, математические буквы и некоторые совместимые символы могут измениться при NFKC. Безопаснее:

1. привести строку к NFC;
2. отдельно применить вашу явную карту эквивалентов: `− → -`, `≤ → <=`, `≥ → >=`, если вам это нужно;
3. сохранить `raw`, `start`, `end`, чтобы можно было показывать ошибку в исходном тексте.

### 6. Разделите “лексический токен” и “синтаксический оператор”

Хорошая архитектура:

```text
input string
  → lexer: массив сырых токенов
  → normalizer: optional canonical/operator aliases
  → parser: Pratt / precedence climbing / shunting-yard
  → AST
  → semantic pass: типы, функции, CAS-правила
```

На стадии `lexer` токен `-` не должен знать, это `UnaryMinus` или `Subtract`. Токен `!` не должен знать, это `Not` или `Factorial`. Токен `^` не должен знать, это `Power` или `Xor`.

### 7. Имплицитное умножение лучше вставлять после лексера

Не пытайтесь распознавать `2x` как один токен. Лучше сначала получить:

```text
2x        → NUMBER("2"), IDENT("x")
2(x+1)    → NUMBER("2"), LPAREN, IDENT("x"), OP("+"), NUMBER("1"), RPAREN
)(        → RPAREN, LPAREN
```

Потом отдельный pass может вставить `IMPLICIT_MUL` между совместимыми соседями:

```text
NUMBER IDENT          → NUMBER OP_IMPLICIT_MUL IDENT
NUMBER LPAREN         → NUMBER OP_IMPLICIT_MUL LPAREN
RPAREN IDENT          → RPAREN OP_IMPLICIT_MUL IDENT
RPAREN LPAREN         → RPAREN OP_IMPLICIT_MUL LPAREN
```

Но не вставляйте его между `f` и `(`, если ваш язык считает `f(x)` вызовом функции.

### 8. Храните operator table в данных, а не в коде

Практичный формат:

```json
{
  "raw": "<=",
  "canonical": "≤",
  "tokenKind": "operator",
  "lexicalRisk": "prefix-conflict",
  "enabled": true
}
```

Так будет проще включать/выключать ASCII-синонимы, CAS-специфичные формы и Unicode-операторы.

## Самые опасные группы для токенизатора

| Группа | Почему опасна | Что делать |
|---|---|---|
| `<`, `>`, `<=`, `>=`, `<=>`, `<->`, `<>`, `<<`, `>>` | Много общих префиксов; ASCII-угловые скобки конфликтуют со сравнениями. | Longest match; Unicode `⟨⟩` для скобок. |
| `!`, `!!`, `!=`, `!==` | `!` может быть NOT или factorial; `!!` может быть double factorial. | Longest match; арность `!` решать позже. |
| `|`, `||`, `‖`, `∣` | OR, модуль, норма, делимость, such-that, условная вероятность. | `|` как общий `BAR`; Unicode-формы по возможности. |
| `-`, `−`, `->`, `<-`, `<->` | Минус, стрелки, похожие Unicode-дефисы. | Явная карта `− → -`; стрелки искать раньше минуса. |
| `:`, `:=`, `::`, `::=`, `:>` | Двоеточие как отношение, such-that, типизация, присваивание. | `:=` искать раньше `:`; роль `:` не решать лексером. |
| `.`, `..`, `...`, десятичные числа | Точка может быть частью числа, оператором, диапазоном, многоточием. | Числа читать до операторов; затем longest match для точек. |
| `^`, `**`, `⊻`, `xor` | Степень против XOR; `**` против `*`. | Не канонизировать `^` без контекста; `**` раньше `*`. |
| Словесные операторы | `and`, `or`, `not`, `in`, `mod` могут быть именами. | Keyword только при границах идентификатора. |

## Рекомендованный набор для MVP

Для первой рабочей версии парсера лучше начать с небольшого, но чистого набора:

```text
Арифметика:      + - * / ^ ** √ !
Сравнения:       = == != < > <= >= ≤ ≥ ≠
Логика:          ¬ ! && || ∧ ∨ -> => ↔ ⇔
Множества:       ∈ ∉ ⊂ ⊆ ⊃ ⊇ ∪ ∩ ∖
Определение:     := ≔
Скобки:          ( ) [ ] { } | |
Разделители:     , ; :
```

После этого можно добавлять Unicode-эквиваленты и редкие relation-символы, не меняя архитектуру лексера.
